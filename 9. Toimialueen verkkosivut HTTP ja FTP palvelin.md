# 9. Toimialueen verkkosivut HTTP ja FTP palvelin
   
 ##   Tehtävänanto

1. Tutustutaan DNS forwarderseihin (DNS-edelleenlähettimet)
2. DNS-välimuistit palvelimella ja asiakkaalla
3. Tietueiden ja käänteisvyöhykkeen lisäys
4. Toimialueem vyöhykkeet ja tietueiden testaus
5. Mielivaltaisten vyöhykkeiden lisäys
6. A- ja PTR-tietueet

## DNS forwarders

<br>

Tehtävässä oli tarkoitus tutustua DNS-edelleenlähettimiin (Forwarder). Aloitin kirjautumalla toimialueeni ohjauspalvelimelle (WindowsServer_DC) ja Server Managerista valitsin: "Tools" -> "DNS". Seuraavaksi valitsin ohjauspalvelimeni nimen alta: "Forwarders". Forwardersien alta (Properties) löytyi Azure Labsin käytössä oleva DNS-forwarder "168.63.129.16". DNS-forwarderin tarkoitus on toimia palvelimena, joka välittää DNS-pyyntöjä eteenpäin muille DNS-palvelimille.

<br>

## DNS-välimuistit

<br>

Tehtävässä oli tarkoitus tutustua DNS-palvelun välimuistin toimintaan verkon osana ja mitä tietoja eri DNS-rooleissa olevat tietokoneet tallentavat. Toimialueellaani ohjauspalvelin (WindowsServer_DC) toimii DNS-palvelimena ja tiedostopalvelin (WindowsServer_FileServer) toimii DNS-asiakkaana. Tärkeää oli, että molemmat virtuaalikoneet olivat päällä harjoituksen aikana.

Testasin aluksi DNS-asiakkaana toimivan tiedostopalvelimen DNS-välimuistia avaamalla ensin tiedostopalvelimen CMD:n ja käyttämällä komentoa:                               
                           
                                      nslookup wikipedia.org

Komento suorittaa suoran haun kyselyn annetusta verkkosivusta. Vastauksena (Non-authorative answer) sain kyseisen sivuston käyttämät IP-osoitteet. Nslookup-komento on hyödyllinen työkalu DNS-palvelun toimivuuden testaamiseen. Vaustauksessa näkyi, että ohjauspalvelimeni toimii nimipalvelimena (asensin aikaisemmassa tehtävässä nimipalvelimen roolin ohjauspalvelimelle). Non-authoritative answer tarkoittaa sitä, että DNS-kysely sai vastauksen jostain muusta kuin verkkotunnuksen auktoritatiivisesta lähteestä (Authoritative Name Server).
<br>

Tämän jälkeen avasin myös selaimella osoitteen: "www.Wikipedia.org", jotta tiedostopalvelin saisi kyseisen domainin nimitedot DNS-asiakaan (tiedostopalvelin) välimuistiin. DNS-välimuistin avulla selaaminen verkossa on nopeampaa ja käytössä oleva verkko kuormittuu vähemmän (Ei tarvitse jatkuvasti lähettää DNS-pyyntöjä). Paikallisen DNS-välimuistin voi tarkistaa CMD:n avulla komennolla: 

      ipconfig /displaydns

   Komento:

      ipconfig /flushdns

   Taas tyhjentää DNS-välimuistin.

   <br>

   Tämän jälkeen suoritin myös aikaisemmat testit ohjauspalvelimella, joka toimii DNS-palvelimena. Nslookup komento onnistui, kuten myös DNS-välimuistin toimivuus. Seuraavaksi siirryin tarkastelemaan DNS-palvelimena toimivan tietokoneen (ohjauspalvelin) DNS-palvelinohjelmiston välimuistia. "Server Manager" -> "Tools" -> "DNS" pääsin DNS Manageriin. Laitettua Advanced-näkymän päälle sain näkyviin kansion "Cached Lookups", missä näkyy DNS-palvelinohjelmiston välimuistiin tallennetut tiedot. Tiedoissa on mm. haettujen eri domain-nimien käyttämiä IPv4- ja IPv6-osoitteita (A- ja AAAA-tietue). DNS-Palvelimen välimuistin pääsee tyhjentämään DNS Managerin kautta tai ylläpitäjän oikeuksin CMD-komennolla:

         dnscmd/clearcache (Mikäli kyseiselle tietokoneelle on asennettu Windows DNS Server-rooli)
<br>



## Tietueiden ja käänteisvyöhykkeen lisäys
   
<br>

Aloitin luomaan kahta alias-nimeä (CNAME) suoran haun vyöhykkeelleni (Forward Lookup Zone), jotka viittaavat tiedostopalvelimeni nimeen. Loin alias-nimet ohjauspalvelimeni DNS Managerissa ensin valitsemalla vyöhykkeeni "partanen.lan" ja sitten "New Alias (CNAME)...". Loin kasi uutta CNAME-tietuetta:

      web.partanen.lan
      intra.partanen.lan

Määrittelemällä ensin Alias-nimen (intra tai web) ja sen jälkeen valitsemalla haun avulla (Browse...) tiedostopalvelimeni.


<img width="1064" height="513" alt="image" src="https://github.com/user-attachments/assets/1df1ef16-f283-442c-b484-e3f74c0bca48" />

> Luodut uudet alias-nimet (CNAME). tiedostopalvelimelle.

<br>

Käänteisen haun vyöhykkeen (Reverse Lookup Zone) lisäsin DNS Managerin kautta kohdasta: "Reverse Lookup Zones" ja "New zone...". Käänteisen haun vyöhykkeiden tehtävä on selvittää IP-osoitetta vastaava domain-nimi, esim. kun suorittaa nslookup-komennon IP-osoitteelle. Käänteisvyöhykkeiden luomiseen oli apuna asennusohjelma, jonka avulla pääsi määrittämään haluamansa asetukset. Valitsin: 

      Zone type:                               Primary zone 
      Store the zone in Active Directory:      Yes
      Zone data replication:                   To all DNS servers running on domain controllers in this domain
      Reverse Lookup Zone name:                IPv4 Reverse Lookup Zone 
      Network ID:                              Hyper-V verkon verkkoavaruuden arvo: 10.208.0
      Dynamic Update:                          Allow only secure dynamic updates

<br>

PTR-tietueita käytetään käänteisissä nimenselvityspyynnöissä kertomaan kohteen nimi. Uudelle käänteisen haun vyöhykkeelle loin PTR-tietueet ohjauspalvelimelle ja tiedostopalvelimelle. valitsin ensin luodun käänteisen haun vyöhykkeen "0.208.10.in-addr.arpa" ja sitten "New Pointer (PTR)...". Host IP Address-kohtaan oli syötettävä halutun tietokoneen IPv4-osoitteen host-osa, minun tapauksessa 10 ja 12.

## A- ja PTR-tietueet


Lisäsin uudet A-tietueet uudelle suoran haun vyöhykkeelle: "PARROI.COM" valitsemalla sen DNS Managerista ja "New Host (A or AAAA)...". Loin seuraavat A-tietueet:



      Name      IP address
      HOST-1    10.120.41.10 
      HOST-2    10.120.41.20 
      HOST-3    10.120.41.30 



<br>

Valitsin myös "Create associated pointer (PTR) record", mikä luo käänteisvyöhykkeen PTR-tietueet automaattisesti. Vastauksena tehtävän lisäkysymykseen: "Tiedätkö, miksi mikään koneista ei ole osoitteessa 10.120.41.1?", yleensä standardin mukaisesti verkon ensimmäinen käytettävissä oleva osoite (tässä tapauksessa 10.120.41.1) on varattu reitittimelle (Default Gateway-oletusyhdyskäytävä).
