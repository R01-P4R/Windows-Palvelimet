# 4. Tiedostopalvelin

## Tehtävänanto

1. Määritellään tiedostopalvelimen runko
2. Käynnistetään ohjauspalvelin ja tehdään alkumääritykset
3. Määritetään ohjauspalvelimen staattinen IP-osoite
4. Määritetään ohjauspalvelimen nimi ja poistetaan lukitusruutu-ominaisuus pois käytöstä
5. Aktivoidaan ohjauspalvelimen käyttöjärjestelmä
6. Asennetaan ohjauspalvelimelle tarvittavat roolit ja ohjelmisto

##  Tiedostopalveliment rungon määritys

Aloitin kirjautumalla Azure Labsiin ja avaamalla virtuaalikoneen. Tässä harjoituksessa luodaan uusi tiedostopalvelin virtuaaliympäristöön. Ensimmäiset askeleet ovat melkein identtiset tehtävä 3. Ohjauspalvelimen kanssa. Aloitin luomaan uutta virtuaalikonetta hyödyntäen aikaisemmassa harjoituksessa luotua virtuaalikovalevyä "WindowsServer_FileServer". Avasin Hyper-V managerin ja valitsin "New -> Virtual Machine. Annoin uudelle virtuaalikoneelle nimeksi "WindowsServer_FileServer" ja valitsin ensimmäisen generaation. Muistin määräksi asetin 3072 MB ja muistin dynaamisen allokoinnin laitoin päälle. Verkkoyhteden adapteriksi valitsin aikaisemmin luodon HyperVVSwitch ja virtuaalikovalevyksi valitsin harjoituksessa "2. Virtuaalikoneiden malli ja palvelinkoneiden alustus" luodun WindowsServer_FileServer.vhdx templaten. Lopuksi vielä muutin Hyper-V managerin kautta käytettyjen virtuaaliprosessorien ytimien määräksi 4.

## Tiedostopalvelimen käynnistys ja alkumääritykset

Aloitin yhdistämällä luotuun tiedostopalvelimeen WindowsServer_FileServer Hyper-V managerin kautta. Koneen käynnistettyä valitsin maa- ja näppäimistö asetuksiksi "Finland" ja "Finnish". Kielen muutin "English (United States)", sillä IT-maailmassa se on ns. de facto-kieli. Seuraavaksi hyväksyin lisenssiehdot, määritin ylläpitäjän salasanan kurssiohjeiden mukaisesti ja kirjauduin virtuaalikoneen työpöydälle. 


## Tiedostopalvelimen staattinen IP-osoite ja sen muuttaminen yksityiseksi

Avasin virtuaalikoneen WindowsServer_FileServer työpöydälle Server Manager-työkalun ja valikosta "Local Server" valitsin Ethernetin vierestä "IPv4 address assigned by DHCP, IPv6 enabled". Sitten valitsin Ethernet-verkkokortista "Properties" ja valitsin "Internet Protocol Version 4 (TCP/IPv4)" Tästä valikosta valitsin vielä "Properties" ja määritin ohjeen mukaisesti IPv4-tiedot (IP-addess, Subnet mask, Default Gateway ja DNS-asetukset) seuraavanlaisesti:

    IP address: 10.208.0.12
    Subnet mask: 255.255.255.0
    Default Gateway: 10.208.0.1
    Preferred DNS sever: 10.208.0.10  <- Ohjauspalvelimen IP-osoite

Nämä asetukset määritettyä sain Windowsilta ilmoituksen uuden verkon luomisesta ja verkkotilatyypin määrityksestä. Tässä tehtävässä määritin sen manuaalisesti yksityiseksi. Aloitin avaamalla cmd:n ylläpitäjänä ja kirjoitin komennon: "gpedit.msc", mikä avaa kyseisen ohjelman, jonka avulla voi määrittää ryhmäkäytäntöjä. Muokkasin kohdasta: "Computer Configuration" -> "Windows Settings" -> "Security Settings" -> "Network List Manager Policies" -> "Network" ja hiiren oikealla "Properties". Valitsin valikosta "Private" ja hyväksyin muutokset. Tarkistin, että virtuaalikone WindowsServer_DC oli päällä taustalla ja suortin cmd:ssä komennon: "nslookup Haaga-helia.fi". Sain ohjeen mukaisen vastauksen, joten oletettavasti yhteydet toimivat ja DNS-palvelimen ip-osoite on oikein.

<br>

Lopuksi tarkastin virtuaalikoneen WindowsServer_FileServer Server Managerista, että palomuuri- ja osoiteasetukset olivat halutut.


## Tiedostopalvelimen nimi ja lukitusruudun poistaminen käytöstä

Aloitin valitsemalla virtuaalikoneen WindowsServer_FileServer Server Managerin ja valikosta Local Server valitsin luodon tietokoneen nimen. Kohdasta "Computer Name" valitsin "Change..." ja määritin ohjeiden mukaisesti uuden nimen "FileServer-Partanen" tiedostopalvelimelle. Nimen muutoksen jälkeen oli virtuaalikoneen käynnistettävä uudestaan.

## Tiedostopalvelimen käyttöjärjestelmän aktivointi

Aloitin avaamalla virtuaalikoneen WindowsServer_FileServer:n komentorivin (cmd) ylläpitäjän oikeuksin ja suoritin komennon: "slmgr /skms kms.core.windows.net" kertoakseni slmgr.vbs-työkalulle missä KMS-aktivointipalvelin sijaitseen. Tämän jälkeen pääsin suorittamaan aktivointipyyntöä komennolla "slmgr /ato", jonka jälkeen sain ilmoituksen, että aktivointi onnistui.



## Tiedostopalvelimen liittäminen toimialueeseen

Aloitin tarkistamalla, että molemmat koneet olivat päällä Azure-virtuaalikoneen Server Managerissa, minkä jälkeen yhdistin virtuaalikoneeseen WindowsServer_FileServer ja Server Managerissa valitsin: "Local Server" ja "WORKGROUP". Kohdasta "Change..." pääsin lisäämään virtuaalikoneen toimialueeseen "partanen.lan". Tämä piti hyväksyä toimialueen, eli ohjauspalvelimen ylläpitäjän tunnuksin. Liittymisen jälkeen oli WindowsServer_FileServer käynnistettävä uudestaan. Lopuksi vielä kokeilin kirjautua toimialueen ylläpitäjän tunnuksin tiedostopalvelin koneelle ja se onnistui.

##  Tiedostopalvelun asentaminen ohjauspalvelimelta




Aloitin avaamalla WindowsServer_DC virtuaalikoneen Server Managerin ja kohdasta "Manage" valitsin "Add Servers".

<br>


<img width="931" height="589" alt="image" src="https://github.com/user-attachments/assets/f2e0e6cb-15e1-4b5d-8a28-f6112a176c74" />

<br>


Active Directory-välilehdeltä hain aikaisemmin luodun tiedostopalvelimen haulla: "FILESEVER" ja lisäsin sen tunnettujen laitteiden listaan. Siirryin takaisin Server Managerin All Servers-näkymään ja valitsin tiedostopalvelimeni. Kohdasta "Manage" -> "Add Roles and Features" pääsin lisäämään tiedostopalvelimen roolit ja ominaisuudet valitsemalla ensin "Role-based or feature-based installation" seuraavalla sivulla luodun tiedostopalvelimeni ja sitten seuraavat roolit: "File Server" ja  "File Server Resource Manager" ("Include management tools" valinta myös avautuvassa ikkunassa) näiden roolien kanssa. Select Features-vaihtoehdot jätin oletukseksi. Viimeisellä sivulla valitsin "Restart the destination server automatically if required" ja painoin "install". Asennuksen lopuksi käynnistin tiedostopalvelimen (WindowsServer_FileServer) uudestaan hyödyntäen ohjauspalvelimen (WindowsServer_DC) Server Manageria.


## Verkkojakone luonti

Aloitin kirjautumalla tiedostopalvelimeen toimialueen ylläpitäjän tunnuksin ja avasin "Server Manager" -> "File and Storage Services".

<br>

<img width="696" height="439" alt="image" src="https://github.com/user-attachments/assets/ee473277-26ba-4a67-a5b8-ec647d3c4adf" />

<br>

Loin uuden jaon (share) asennusohjelman avulla. Valitsin ensin "SMB Share – Quick" ja seuraavassa kohdassa määritin uuden polun "C:\Files", mikä luo mnyös tarvittavan kansion, jos sitä ei vielä ole. Verkkolevyjaolle annoin nimeksi "Data" ja valitsin seuraavaksi "Allow caching of share". Seuraavaksi pääsin tarkastelemaan luotavan verkkojaon oikeuksia ja yhteenvetoa valituista asetuksista hyväksyin ne muuttamatta ja painoin "install" ja "close". 

<br>

Siirryin ohjauspalvelimen Server Manageriin seuraavaksi ja kohdasta "All Servers" valitsin tiedostopalvelimen ja "Computer Management". Sain ilmoituksen, että tiedostopalvelimen palomuuriasetukset estävät pääsyn Event Viewerille. Siirryin takaisin tiedostopalvelimelle ylläpitäjän tunnuksin ja muutin: "Windows Defender Firewall with Advanced Security" -> "Inbound Rules" seuraavat säännöt hyväksytyksi: Remote Event Log Management (NP-In), Remote Event Log Management (RPC) ja Remote Event Log Management (RPC-EPMAP). Tämä avaa tiedostopalvelimen portit ohjauspalvelimen työkaluille.

Asetuksia muuttamalla pääsin ohjauspalvelimelta yhdistämään etänä tiedostopalvelimelle ja tarkastamaan sen hallintanäkymää, sekä tiedostoja. Seuraavaksi loin ohjauspalvelimelta käsin uuden jaon tiedostopalvelimelle avaamalla ensin Server Managerin ja tiedostopalvelimen kohdalta: "Computer Management". Polussa "Shared Folders" -> "Shares" -> "New Share..." pääsin luomaan uutta jakoa asennusohjelman avulla. Loin uuden kansion "C:\MManagement" aikaisemmin luodon kansion "Files" alapuolelle ja jaon nimeksi annoin "Person" asetuksella: "No files or programs from the shared folder are available offline". Lopuksi määritin vielä verkkojaolle vähimpien oikeuksien periaatteella "Administrators have full access; other users have no access". Uuden jaon luominen onnistui.
